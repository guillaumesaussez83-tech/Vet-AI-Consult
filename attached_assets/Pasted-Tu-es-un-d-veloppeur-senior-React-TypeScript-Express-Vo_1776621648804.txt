BUG-02 [CRITIQUE - 1h] : "Finaliser et facturer" → 400 
======================================================
SYMPTÔME : Le bouton "Finaliser et facturer" envoie un 
PATCH /api/consultations/:id avec { statut: 'terminee', actes: [...] }. 
Le backend rejette la requête avec 400 "Données invalides" car 
le schéma Zod du PATCH ne contient pas le champ "actes".

SOLUTION CÔTÉ BACKEND (server/routes/consultations.ts) :
Option A (recommandée) : Dans le handler PATCH /api/consultations/:id,
extraire le champ "actes" du body avant validation Zod, puis :
  1. Mettre à jour le statut via updateConsultationSchema (sans actes)
  2. Si actes est présent et non vide, créer les actes via insertActeSchema
  3. Créer automatiquement la facture via POST /api/consultations/:id/facture
     avec montantHT et montantTTC calculés depuis la somme des actes

Option B : Modifier le schéma Zod pour accepter le champ actes 
(optionnel) et l'ignorer dans la validation, en gérant les actes 
dans un second appel.

SOLUTION CÔTÉ FRONTEND (si Option B) : Séparer le PATCH statut 
et la création d'actes en deux appels distincts dans le handler 
du bouton "Finaliser et facturer".

BUG-04 [MINEUR - 15 min] : Mode paiement affiché en raw
======================================================
SYMPTÔME : Dans les listes Facturation et Encaissements, 
le mode de paiement s'affiche en snake_case brut 
("carte_bancaire") au lieu du libellé formaté 
("Carte bancaire"). La page de détail d'une facture affiche 
correctement "Carte bancaire (puce)" — la logique de formatage 
existe donc déjà, mais n'est pas appliquée dans les listes.

FIX : Créer ou réutiliser un helper formatModePaiement() :

const formatModePaiement = (mode: string): string => {
  const labels: Record<string, string> = {
    'carte_bancaire': 'Carte bancaire',
    'especes': 'Espèces',
    'cheque': 'Chèque',
    'virement': 'Virement',
    'autre': 'Autre'
  };
  return labels[mode] || mode;
};

Appliquer ce helper partout où le mode de paiement est affiché 
en liste (composants FactureList, EncaissementList, etc.).

BUG-06 [MINEUR - 1h] : Top 10 actes affiche "—" 
dans les Statistiques
======================================================
SYMPTÔME : Le widget "Top 10 actes & produits" affiche 
"—" pour le nom de l'acte au lieu du nom réel 
(ex: "Consultation", "Vaccination", etc.).

CAUSE PROBABLE : La requête SQL/ORM d'agrégation pour les stats 
ne joint pas correctement la table des actes ou le champ 
"description" n'est pas remonté dans le résultat.

LOCALISATION : server/routes/statistiques.ts (ou équivalent), 
requête d'agrégation TOP actes.

FIX : Vérifier la requête de top actes. Elle devrait grouper 
par "description" (le libellé de l'acte) et non par "acteId" 
(qui peut être null pour les actes libres). Exemple :

SELECT description, COUNT(*) as nb, SUM(prix_unitaire * quantite) as ca
FROM actes 
WHERE consultation_id IN (SELECT id FROM consultations WHERE ...)
GROUP BY description
ORDER BY nb DESC
LIMIT 10
